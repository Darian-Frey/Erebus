## #11 — Starfield diffraction spikes clipped at grid-cell boundaries — 2026-04-26 — fixed

**Symptom.** First successful web render showed the nebula correctly but stars appeared in visibly clumped patches with hard gaps between them — a "groupy" gridded look unlike the smooth random distribution we got at low densities during Phase 4.

**Root cause.** `star_layer()` in [shaders/nebula/raymarch.wgsl](../shaders/nebula/raymarch.wgsl) hashed the *single* cube cell that `floor(dir * scale)` falls into, evaluated the star generated by that cell, and returned. With density 80 the cell angular width is ~1/80 ≈ 0.0125 rad, while `spike_length` defaults to 0.012 rad — so any star sitting near a cell boundary had one or both of its diffraction-spike arms truncated where the spike crossed into a neighbouring cell. The gridded discontinuity reads as clustering.

**First attempt — rejected.** Loop over the 3×3×3 neighbourhood (27 cells per layer). Two failure modes:

1. **Performance** dropped to ~0.3 fps (3 s/frame) — 27 × 3 layers = 81 hashes per pixel + 81 conditional texture samples on the diffuse path.
2. **Visual regression**: `floor(dir * scale)` lands on a thick shell of cells at radius ≈ scale, so the radial neighbours `(cell.x, cell.y, cell.z ± 1)` `normalize()` to nearly the same direction as `cell` itself. We were drawing 2–3 *duplicate* stars at every cell, producing huge spherical "blobs" tiled across the sphere.

**Fix shipped.** Reverted to single-cell evaluation. Instead, clamp `spike_length` to `min(starfield.spike_length, 0.5 / scale)` so each star's diffraction cross fits inside its home cell by construction. The user-facing `spike_length` slider still works for low densities; at high densities it caps at half a cell width and the "grouping" cannot occur. Future improvement: octahedral 2D mapping would give true uniform sphere coverage with a 2D 3×3 neighbour loop (9 cells, no radial duplicates).

**Lesson.** A 3D shell-of-cells parametrisation of a unit sphere has hidden duplicate neighbours along the radial axis — *do not* use a 3D box neighbourhood loop on `floor(dir * scale)`. Either keep evaluation single-cell (and constrain the kernel) or change the parametrisation to one that's natively 2D on the sphere.

**Follow-up (2026-04-27).** Replaced the 3D cube hash with **octahedral 2D mapping** (Cigolle et al. 2014) in [shaders/nebula/raymarch.wgsl::dir_to_oct / star_layer](../shaders/nebula/raymarch.wgsl). The unit direction now projects to a [-1, 1]² square via L1-norm + corner unfold; cells hash on (u, v, 0). Solid-angle distortion drops from ~3× (cube) to ~1.5× at the worst (square corners), with no axis alignment, so the visible "constellation" clustering at high yaw/pitch is gone. Single-cell evaluation kept (no neighbour search) — same per-pixel cost as the original cube-grid version but uniform sphere coverage. Star layer hashes are now properly independent across the 3 parallax octaves.

## #10 — `textureSample` inside non-uniform control flow rejected by WebGPU — 2026-04-26 — fixed

**Symptom.** Native build worked fine; web build rendered the canvas opaque-black after a brief orange flash from the diagnostic shader. Both `prepare()` and `paint()` callbacks ran every frame, but the `nebula_pipeline` was silently invalid. DevTools console eventually surfaced:

```text
Error while parsing WGSL: 'textureSample' must only be called from uniform control flow
        let albedo_color = textureSample(gradient_tex, gradient_sampler, lut_t).rgb;
note: control flow depends on possibly non-uniform value
        if (transmittance < nebula.transmittance_cutoff) { ... }
```

The errors flooded the console (one per frame) but didn't hit our `console.error` interception path, so the page-level diagnostic banner showed nothing.

**Root cause.** WebGPU's spec strictly requires implicit-LOD `textureSample` to be invoked only from *uniform control flow* — i.e., not after a branch / loop where divergent threads might disagree. Our nebula raymarch uses an early-out `if (transmittance < cutoff) break;` which makes everything after it non-uniform; the gradient-LUT lookup `textureSample(gradient_tex, ...)` further down hit that rule. Native wgpu (Vulkan/Metal/DX12) tolerates this because the underlying backends do; Chrome's WebGPU enforces the spec strictly.

**Fix.** Switched the in-loop sample to [`textureSampleLevel`](../shaders/nebula/raymarch.wgsl) (explicit LOD = 0.0) which doesn't need derivatives and so doesn't require uniform control flow. Did the same prophylactically in [composite.wgsl](../shaders/composite.wgsl) for the HDR + bloom mip-0 reads — they happen at root level today but the cost of `textureSampleLevel` vs `textureSample` is identical for non-mipmapped 1D/2D textures (both bypass the LOD calculation), so future-proof.

**Lesson.** WebGPU is the strictest of the wgpu backends. Anywhere `textureSample` is inside a `for`/`while`/`if` that depends on a per-pixel value, switch to `textureSampleLevel` (or `textureSampleGrad`) preemptively. It's the same cost on the GPU and works on every backend.

## #9 — Chrome's WebGPU rejects wgpu 0.20's `maxInterStageShaderComponents` limit — 2026-04-26 — fixed (Phase 9.5)

**Symptom.** WASM bundle builds and loads, but Chrome 130+ throws on `requestDevice`:

```text
OperationError: Failed to execute 'requestDevice' on 'GPUAdapter':
The limit "maxInterStageShaderComponents" with a non-undefined value is not recognized.
```

The on-page panic banner shows it; live preview is blank.

**Root cause.** `maxInterStageShaderComponents` was removed from the WebGPU spec around 2023 (renamed to `maxInterStageShaderVariables`). Our pinned `wgpu = "0.20"` (via `eframe = "0.28"`) still includes the obsolete field name in the JS object it builds for `requestDevice`. Chrome's current WebGPU implementation strictly rejects unknown limit keys. Firefox + Safari likely behave the same way; this is a spec-conformance issue, not a Chrome quirk.

**Fix.** Phase 9.5 dependency upgrade — `eframe 0.28 → 0.30`, `wgpu 0.20 → 23`, `egui 0.28 → 0.30`, `egui-wgpu 0.28 → 0.30`. The migration touched:

- `wgpu::RenderPipelineDescriptor` and `ComputePipelineDescriptor` got a `cache: Option<&PipelineCache>` field back — added `cache: None` to all 8 pipeline descriptors.
- `entry_point` is now `Option<&str>` (was `&str`) — wrapped all entry-point strings in `Some(...)`.
- `egui_wgpu::CallbackTrait::paint` reverted to a non-generic signature with `RenderPass<'static>` and a non-lifetimed `&CallbackResources` (the 0.28 generic-lifetime version was the outlier).
- `eframe::WebRunner::start` now takes `HtmlCanvasElement` directly instead of a `&str` canvas-id — the lib resolves the element via `web_sys::window().document().get_element_by_id(...)` and downcasts.
- `egui_wgpu::WgpuConfiguration` lost its top-level `power_preference` field — defaults are fine; Chrome's WebGPU adapter discovery on Linux is no longer brittle in this stack.
- `egui::ComboBox::from_id_source` deprecated in favour of `from_id_salt` — bulk-renamed.

Result: 2.9 MB raw / **1.3 MB gzipped** wasm bundle (smaller than the 0.28 build), wgpu requests valid limits, Chrome WebGPU accepts `requestDevice` cleanly.

**Lesson.** When pinning a dep to a specific version, watch for upstream-spec changes the dep follows on a different release cadence than yours. WebGPU is still in motion and any wgpu pin older than ~6 months risks runtime drift on the latest browsers even if the build is clean.

**Workaround for users today.** Use the native desktop build. The desktop binary's wgpu path goes through Vulkan / Metal / DX12 directly, never touching the JS WebGPU layer that has the spec drift.

**Lesson.** When pinning a dep to a specific version, watch for upstream-spec changes the dep follows on a different release cadence than yours. WebGPU is still in motion and any wgpu pin older than ~6 months risks runtime drift on the latest browsers even if the build is clean.

## #8 — Starfield Kelvin sliders had no perceptible effect — 2026-04-26 — fixed

**Symptom.** Setting `T_min` and `T_max` to extreme values (both very low expecting red, both very high expecting blue) produced visually identical exports — stars stayed pale-blue/white in both cases. The user noticed and flagged it.

**Root cause.** The starfield shader keyed both `magnitude` and `temperature` off the same hash component `h.y`:

```wgsl
let mag = pow(h.y, starfield.imf_exponent);
let temp = mix(starfield.temperature_min, starfield.temperature_max, h.y);
```

Coupled in this way, bright stars (high `h.y`) always pinned to `T_max` and dim stars (low `h.y`) always pinned to `T_min`. Because the IMF exponent (default 5) biases the field toward dim sub-pixel stars, the only *visible* stars were the ~5 % brightest — which were all locked to `T_max`. Result: `T_min` was effectively ignored, and the field was monochromatic at any single moment.

**Fix.** Sample a second hash with a fixed integer offset for the temperature roll, in [shaders/nebula/raymarch.wgsl](../shaders/nebula/raymarch.wgsl):

```wgsl
let temp_h = hash3i(cell + vec3<i32>(13, 17, 23), frame.seed ^ layer_seed);
let temp = mix(starfield.temperature_min, starfield.temperature_max, temp_h.x);
```

Cost: one extra PCG3D per star (negligible). Visual benefit: red giants and blue dwarfs at all magnitudes; both Kelvin sliders now drive a clearly visible response.

**Lesson.** Whenever two random outcomes are sampled from the same hash channel, ask whether the correlation is intentional (e.g. mass-luminosity for *physically* realistic stars) or accidental (e.g. accidentally fused-domain-hashing). Here a *more* physically realistic version (mass-temperature coupling) was less *visually* useful because the IMF skews the visible population to one end.

## #7 — RON serializes Rust fixed-size arrays as tuples, not lists — 2026-04-26 — fixed

**Symptom.** All three shipped preset RONs failed to load with `Expected opening '('` at the position of `lights: [...]`. The unit tests `shipped_presets_load` and `shipped_presets_round_trip` exposed this immediately.

**Root cause.** I'd written the `[PointLight; 4]` array as a JSON-style list `[a, b, c, d]`. RON serializes Rust *fixed-size arrays* as tuples — `(a, b, c, d)` — and only `Vec<T>` as lists. The `gradient: Vec<GradientStop>` field is correctly a list; `lights: [PointLight; 4]` must be a tuple.

**Fix.** Changed the `lights:` field's brackets in [synthwave.ron](../assets/presets/synthwave.ron), [cyberpunk.ron](../assets/presets/cyberpunk.ron), [retro_scifi.ron](../assets/presets/retro_scifi.ron) from `[...]` to `(...)`.

**Lesson.** When hand-writing RON for any Rust struct, sketch a tiny `ron::ser::to_string_pretty(&default_value, ...)` first and copy the format. Don't assume JSON-list syntax is interchangeable with RON tuple syntax — the distinction matters when the Rust type is a fixed-size array vs a `Vec`.

## #6 — WGSL rejects unary `+` in expressions — 2026-04-26 — fixed

**Symptom.** `cargo test --test wgsl_validation` failed with `error: expected expression, found '+'` at the AgX polynomial in [shaders/composite.wgsl](../shaders/composite.wgsl):

```wgsl
return + 15.5 * x4 * x2 - 40.14 * x4 * x + ...;
```

**Root cause.** WGSL only supports unary `-`, not unary `+`. The leading `+` was a copy-paste from the GLSL/HLSL Three.js AgX reference, which both accept it.

**Fix.** Dropped the leading `+` and added an explicit `vec3<f32>(0.00232)` in the trailing constant term so naga's vector arithmetic could resolve the type. Same lesson for any future shader port from GLSL: scan for `return + …` and `return -<scalar>` patterns.

**Lesson.** When porting math from GLSL/HLSL to WGSL, run the file through `cargo test --test wgsl_validation` immediately rather than relying on the runtime device validation — naga's error messages point straight at the offending line.

## #5 — Per-frame IGN rotation reads as flicker without TAA — 2026-04-26 — fixed

**Symptom.** With Phase 3.5 baked-noise speedup running at ~40 fps, the user reported the nebula was visibly flickering frame-to-frame on a static scene.

**Root cause.** Phase 2's `ign(pixel, frame_index)` rotated the interleaved-gradient noise tile per frame so that residual sampling artifacts would smear into temporal noise. The original assumption was that downstream bloom + tonemap (Phase 5) would smooth the rotation into perceptual mush. Without bloom present, the rotation IS the flicker — every frame's first-sample jitter offset shifts by ~1 step length, so the band of pixels that catches a thin density spike changes visibly each frame. At 40 fps with a static scene the temporal aliasing is unmistakable.

**Fix.** Made `ign()` purely spatial in [shaders/nebula/raymarch.wgsl](../shaders/nebula/raymarch.wgsl) — dropped the `frame_index` argument. The temporal jitter will be re-introduced in Phase 5 alongside proper TAA-style accumulation where bloom hides the residual.

**Lesson.** Don't ship temporal aliasing tricks without the temporal smoothing they were designed against. "It'll get smeared by bloom later" is a valid argument *only after bloom exists* — until then, static jitter is correct.

## #3 — `Receiver<notify::Event>` is `Send` but not `Sync` — 2026-04-25 — fixed

**Symptom.** `cargo check` errored: `std::sync::mpsc::Receiver<notify::Event> cannot be shared between threads safely` for the renderer stored in `egui_wgpu::CallbackResources`.

**Root cause.** `egui_wgpu::CallbackResources` is `type_map::concurrent::TypeMap`, which requires `Any + Send + Sync`. `mpsc::Receiver` only satisfies `Send`.

**Fix.** Wrapped the receiver in `std::sync::Mutex` inside [src/render/hot_reload.rs](../src/render/hot_reload.rs). `poll()` calls `self.rx.get_mut()` since we already have `&mut self` and don't need a runtime lock.

**Lesson.** Anything that lives inside `CallbackResources` must be `Send + Sync`. `Mutex` is the standard upgrade for `Send`-only channel ends.

## #2 — `paint()` lifetime mismatch with `egui_wgpu::CallbackTrait` — 2026-04-25 — fixed

**Symptom.** `cargo check` errored: `method not compatible with trait: lifetime mismatch`. The trait expected `fn paint<'a>(&'a self, .., &mut RenderPass<'a>, &'a CallbackResources)`; our impl had `&mut RenderPass<'static>` and no `'a`.

**Root cause.** Misread the egui-wgpu 0.28 trait signature. The pass lifetime is tied to `&self` and `&CallbackResources`, not `'static` — that's how the trait threads borrows of resources into the pass safely.

**Fix.** Restored the `<'a>` generic on both `NebulaCallback::paint` ([src/gui/mod.rs](../src/gui/mod.rs)) and `ErebusRenderer::composite` ([src/render/graph.rs](../src/render/graph.rs)).

**Lesson.** When a trait uses a generic lifetime, the impl must repeat it verbatim. Don't substitute `'static` "to simplify" — variance flips and the compiler rejects it.

## #1 — winit fails to compile under eframe with `default-features = false` — 2026-04-25 — fixed

**Symptom.** First `cargo check` produced 65 errors inside `winit-0.29.15`, starting with `The platform you're compiling for is not supported by winit` and a cascade of `unresolved import self::platform` / `type annotations needed`.

**Root cause.** Disabling eframe's default features stripped the `wayland` and `x11` features it forwards to winit. winit then has no platform backend selected on Linux and refuses to compile.

**Fix.** Added `"wayland", "x11"` to the eframe feature list in [Cargo.toml](../Cargo.toml).

**Lesson.** When trimming default features off a windowing crate, always re-enable the platform backends explicitly. eframe's relevant features are `wayland`, `x11`, `glow`, `wgpu`, `accesskit`, `persistence`.
